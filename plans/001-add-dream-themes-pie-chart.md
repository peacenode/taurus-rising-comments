## Why this matters

The current page makes the placement patterns easy to scan but has no visual
summary of what Taurus-rising respondents dream about. Add one pie chart for
the seven proposed Dream themes while preserving the existing six placement
graphs, their filtering behavior, search, and response list.

The themes overlap, so the pie must represent the distribution of **theme
assignments**, not mutually exclusive people. If one response is assigned
Home / Belonging, Cultivation, and Stewardship, it contributes one assignment
to each of those three slices. The interface must explain that distinction
next to the chart.

## Product decisions that are settled

1. **This is additive only.** Keep the existing placement graph block and add
   the pie chart beneath its existing methodology note, before search.
2. **Do not change the current placement graph.** Preserve `#stats`, all six
   `STAT_DEFS`, `dist`, `renderStats`, its click listener, and the current
   placement-filter semantics.
3. **Use a true pie chart**, not the previously proposed radial/spoke chart,
   donut, or chord diagram.
4. **Slices show share of all theme assignments.** They do not show percent of
   respondents and must not be labeled that way.
5. **Theme membership is authored backend data.** Never derive it from search
   keywords in browser JavaScript or at build time.
6. **Theme descriptions appear once.** Put the seven descriptions in the
   chart legend/key. Do not add descriptions or classification reasons to
   response rows.
7. **The chart is non-interactive in this phase.** Do not make slices or legend
   rows filter responses. Existing placement filtering remains the only chart
   filtering behavior.
8. **Preserve the current static architecture.** A custom static SVG chart is
   a visualization, not a new UI-control primitive, so this phase does not
   require React, shadcn installation, npm, or a framework migration.
9. **Defer the rest.** Whole Sign/Placidus separation, Career analysis, theme
   chips, theme filtering, respondent normalization, new navigation, and the
   Sagittarius comparison are all out of scope.

## Backend theme model

Keep the source data and editorial analysis separate:

- `data/extracted.json` remains the unchanged source-comment dataset.
- `data/dream_themes.json` defines the seven themes, display order, fixed
  neutral chart color, public description, and private classification
  guidance.
- `data/dream_theme_assignments.json` assigns reviewed theme IDs to stable
  source-response identities. It is the only source of pie membership.
- `data/dream_theme_review.json` records a blind secondary pass and every
  resolved disagreement. It is QA metadata and is never rendered.
- `build_page.py` validates those three files, computes aggregate counts, and
  renders only the public summary into `index.html`.

### Stable response identity

Key every assignment by the pair `(username, created_time)`. That pair is
unique for all 355 current rows. Do not use array indexes because new comments
are inserted newest-first, and do not key only by username because four users
currently have more than one comment row.

For stale-review detection, compute `source_digest` as SHA-256 over the UTF-8
bytes of this canonical JSON serialization:

```python
json.dumps(
    {"dreams": source_row["dreams"], "text": source_row["text"]},
    ensure_ascii=False,
    sort_keys=True,
    separators=(",", ":"),
)
```

Any change to the Dream summary or verbatim text must invalidate the reviewed
assignment until it is checked again.

## Pie-chart semantics

The aggregate is intentionally assignment-based:

```text
theme_count(theme) = number of reviewed assignments containing theme
total_theme_assignments = sum of the seven theme counts
slice_percent(theme) = theme_count(theme) / total_theme_assignments * 100
```

A multi-label response contributes once to each assigned theme. It still
counts only once in `themed_response_count`.

The generated section must show:

- heading: `Dream themes`;
- summary: `<themed response count> of <reviewed Dream count> reviewed Dream responses map to at least one theme · <total theme assignments> theme assignments`;
- methodology: `A response can belong to more than one theme. Pie slices show the share of all theme assignments, not the share of people.`;
- a full pie chart, not a donut;
- an ordered legend with swatch, label, assignment count, percentage of all
  assignments, and the one public description;
- zero-count themes in the legend with `0 · 0%`, but no zero-area SVG path.

The percentages in the legend should be shown to one decimal place and may sum
to 99.9% or 100.1% because of display rounding. The underlying unrounded slice
angles must sum to exactly 360 degrees.
